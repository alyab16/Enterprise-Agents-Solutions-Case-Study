\section{Architecture Overview}

The solution implements an \textbf{AI-powered automation agent} that orchestrates enterprise customer onboarding from deal closure through SaaS provisioning. Built with \textbf{Pydantic AI} for agentic tool calling and \textbf{FastAPI} for the REST interface, the agent uses native LLM tool calling to autonomously reason through 24 registered tools. Each tool is mirrored as a \textbf{FastMCP} server definition for MCP-style extraction. A second \textbf{CS Assistant Agent} shares all tools but returns free-form text for conversational interaction via a \textbf{Streamlit UI} with four pages (Dashboard, Portfolio Overview, Run Onboarding, Chat). Data flows sequentially across systems via \textbf{cross-system chaining}: Opportunity.ContractId $\rightarrow$ SF Contract.Id $\rightarrow$ CLM.salesforce\_contract\_id $\rightarrow$ Invoice.clmContractRef, with fallback to account-based lookups when chain links are missing.

\section{AI Agent Application}

The agent leverages \textbf{OpenAI GPT-4o-mini} (configurable via \texttt{OPENAI\_MODEL}) with \textbf{Pydantic AI} for native tool calling. The LLM autonomously decides which of the 24 registered tools to call, in what order, and how many times. Falls back to \textbf{Ollama} local models when no API key is available. A second \textbf{CS Assistant Agent} shares all tools with free-form text output for conversational monitoring via the Streamlit UI.

\vspace{0.05em}
\begin{minipage}[t]{0.56\textwidth}
\vspace{0pt}
\subsection{LLM-Powered Intelligence}
\begin{itemize}
\item \textbf{Risk Analysis}: Evaluates API errors, violations, warnings $\rightarrow$ risk levels (low/medium/high/critical) with business impact and resolution time estimates.
\item \textbf{Summary Generation}: Human-readable reports for CS teams, translating technical states to actionable insights.
\item \textbf{Action Recommendations}: Prioritized steps with owner assignment (CS, Finance, IT/DevOps, Legal, Sales Ops).
\item \textbf{Error Interpretation}: Converts technical codes (e.g., \texttt{INVALID\_SESSION\_ID}) to plain English with resolution steps.
\end{itemize}

\vspace{4pt}

\subsection{Autonomous Actions}
\begin{enumerate}[label=\textbf{(\arabic*)}, itemsep=0pt, topsep=3pt]
\item Auto-provisions the SaaS tenant with tier-appropriate configuration (Enterprise/Growth/Starter).
\item Creates a 14-task onboarding checklist with dependencies, ownership (system/cs\_team/customer), and due dates.
\item Sends Slack notifications to \texttt{\#cs-onboarding-alerts} (blocked) or \texttt{\#cs-onboarding} (success/escalate).
\item Sends customer-facing welcome emails with tenant ID and login URL.
\item Generates HTML/Markdown/JSON reports for auditing and traceability.
\item Post-provisioning: monitors progress, detects risks, sends reminders, escalates stalled onboardings.
\item Portfolio management: aggregated alerts across accounts, suggested actions with approve/dismiss, batch reminders.
\end{enumerate}
\end{minipage}

\item Invoice payment status is current
\item Cross-currency comparison uses historical ECB rates from invoice date with 2\% threshold

\item One Opportunity per Account (no multi-deal)

\end{itemize}
}

\begin{minipage}[t]{0.48\textwidth}
\vspace{0pt}
\subsection{Security Patterns}
{\scriptsize
All integrations are mocked but demonstrate production security patterns:
\begin{itemize}[leftmargin=1.2em]
\item \textbf{OAuth Simulation}: Token expiry and refresh flows with distinct error types for expired vs invalid tokens
\item \textbf{Permission Checking}: Role-based access control verified before every API call
\item \textbf{Credential Validation}: Structured error hierarchy (\texttt{AuthenticationError}, \texttt{AuthorizationError}, \texttt{ValidationError}, \texttt{RateLimitError})
\item \textbf{Audit Logging}: Correlation IDs propagated end-to-end for traceability across all systems
\item \textbf{Error Masking}: No sensitive data (tokens, secrets) exposed in API responses or reports
\end{itemize}
}

\section{Multi-Agent Architecture via Model Context Protocol (MCP) Server Connections}

\begin{minipage}[t]{0.54\textwidth}
\vspace{0pt}
{\scriptsize\textit{Note: All 24 agent tools are mirrored as FastMCP server definitions in \texttt{app/mcp/} (7 servers: Salesforce, CLM, NetSuite, Currency, Provisioning, Notifications, Validation). These can be extracted to standalone MCP services for multi-agent collaboration.}}

\vspace{0.1em}
{\scriptsize
The architecture supports multi-agent collaboration through the \textbf{Model Context Protocol (MCP)}, enabling specialized agents to interact through standardized, governed tool interfaces rather than direct API access. Each integration is wrapped as a FastMCP server, cleanly separating decision logic from execution concerns.
}

\vspace{0.1em}
\begin{table}[H]
\scriptsize
\begin{tabularx}{\textwidth}{@{}lX@{}}
\toprule
\textbf{Agent} & \textbf{Responsibility \& MCP Tools} \\
\midrule
Coordinator & Orchestrates end-to-end workflow (\texttt{salesforce.*}, \texttt{provision.*}) \\
Contract Agent & Monitors contract execution and signatures (\texttt{clm.get\_contract}) \\
Finance Agent & Evaluates billing, payments, and cross-currency alignment (\texttt{netsuite.*}, \texttt{currency.*}) \\
Task Monitor & Detects overdue onboarding tasks (\texttt{tasks.get\_overdue}) \\
\bottomrule
\end{tabularx}
\end{table}

\vspace{-15pt}
{\scriptsize\textbf{Benefits}: Independent versioning per integration $\bullet$ Clear domain ownership $\bullet$ Centralized access control and audit logging $\bullet$ Scalable agent complexity $\bullet$ Cross-agent context sharing.}

\vspace{0.25em}
{\scriptsize
\textbf{Concurrency \& API Execution Strategy}: MCP servers act as controlled execution layers for external APIs, enabling concurrent and batch-aware request handling. Batch or composite API calls are leveraged where supported (e.g., Salesforce composite reads), with fallback to bounded parallel single-entity requests. Concurrency limits, rate limiting, and circuit breakers are enforced at the MCP layer to prevent downstream overload and cascading failures.
}

%-------------------------------
% Agent Runtime & API Governance
%-------------------------------
\vspace{0.2em}
{\small\bfseries\color{primary} Agentic Execution \& API Governance}

\vspace{0.05em}
{\scriptsize
\textbf{Deterministic Decision Boundaries}: The LLM reasons through tools via native tool calling (Pydantic AI \texttt{@agent.tool} decorators), but deterministic business rules enforced by \texttt{validate\_business\_rules} and \texttt{check\_financial\_alignment} define all violations and warnings. The LLM routes (\texttt{PROCEED}/\texttt{ESCALATE}/\texttt{BLOCK}) based on these outputs, ensuring predictable, testable, auditable outcomes.

\vspace{0.1em}
\textbf{Cross-System Data Chaining}: Fetch account first, then user/opportunity in parallel. Sequential chain: Opportunity.ContractId $\rightarrow$ SF Contract $\rightarrow$ CLM (via salesforce\_contract\_id) $\rightarrow$ Invoice (via clmContractRef). Fallback to account-based lookups if chain links are missing. Production optimization: \textbf{(1)} Batch APIs (Salesforce Composite); \textbf{(2)} Concurrent fetches; \textbf{(3)} Bounded exponential backoff (max 3 retries, 1s/2s/4s).

\vspace{0.1em}
\textbf{Idempotency}: Each run keyed by \texttt{correlation\_id} propagated across all systems --- enables safe replays, deduplication, and re-entrant execution without duplicate side effects.

\vspace{0.1em}
\textbf{Error Classification \& Retry}: Retryable (5xx, rate limits) $\rightarrow$ bounded exponential backoff. Non-retryable (validation, auth) $\rightarrow$ immediate halt with diagnostics. Circuit breakers prevent cascading failures.

\vspace{0.1em}
\textbf{Rate Limiting}: Per-source rate limits and queue-based ingestion for webhook traffic spikes. Per-integration concurrency limits prevent API overload.

\vspace{0.1em}
\textbf{Agent Versioning}: Every run records agent version, rule set, and prompt version for auditability, regression analysis, and controlled rollouts (canary deployments).

\vspace{0.1em}
\textbf{LLM Data Governance}: Field-filtered and sanitized data sent to LLMs. PII masked prior to inference. Credentials never exposed beyond integration boundaries.

\vspace{0.1em}
