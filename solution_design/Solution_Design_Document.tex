\section{Architecture Overview}

The solution implements an \textbf{AI-powered automation agent} that orchestrates enterprise customer onboarding from deal closure through SaaS provisioning. Built with \textbf{Pydantic AI} for agentic tool calling and \textbf{FastAPI} for the REST interface, the agent uses native LLM tool calling to autonomously reason through 21 registered tools. Each tool is mirrored as a \textbf{FastMCP} server definition for MCP-style extraction. A second \textbf{CS Assistant Agent} shares all tools but returns free-form text for conversational interaction via a \textbf{Streamlit UI}. Data flows sequentially across systems via \textbf{cross-system chaining}: Opportunity.ContractId $\rightarrow$ SF Contract.Id $\rightarrow$ CLM.salesforce\_contract\_id $\rightarrow$ Invoice.clmContractRef, with fallback to account-based lookups when chain links are missing.

\section{AI Agent Application}

The agent leverages \textbf{OpenAI GPT-4o-mini} (configurable via \texttt{OPENAI\_MODEL}) with \textbf{Pydantic AI} for native tool calling. The LLM autonomously decides which of the 21 registered tools to call, in what order, and how many times. Falls back to \textbf{Ollama} local models when no API key is available. A second \textbf{CS Assistant Agent} shares all tools with free-form text output for conversational monitoring via a \textbf{Streamlit UI} (Dashboard, Run Onboarding, Chat).

\vspace{0.1em}
\begin{minipage}[t]{0.56\textwidth}
\vspace{0pt}
\subsection{LLM-Powered Intelligence}
\begin{itemize}
\item \textbf{Risk Analysis}: Evaluates API errors, violations, warnings $\rightarrow$ risk levels (low/medium/high/critical) with business impact and resolution time estimates.
\item \textbf{Summary Generation}: Human-readable reports for CS teams, translating technical states to actionable insights.
\item \textbf{Action Recommendations}: Prioritized steps with owner assignment (CS, Finance, IT/DevOps, Legal, Sales Ops).
\item \textbf{Error Interpretation}: Converts technical codes (e.g., \texttt{INVALID\_SESSION\_ID}) to plain English with resolution steps and responsible teams.
\end{itemize}

\vspace{8pt}

\subsection{Autonomous Actions}
Upon decision, the agent performs the following autonomous actions:
\begin{enumerate}[label=\textbf{(\arabic*)}, itemsep=0pt, topsep=6pt]
\item Auto-provisions the SaaS tenant with tier-appropriate configuration (Enterprise/Growth/Starter).
\item Creates a 14-task onboarding checklist with dependencies, ownership (system/cs\_team/customer), and due dates.
\item Sends Slack notifications to \texttt{\#cs-onboarding-alerts} (blocked) or \texttt{\#cs-onboarding} (success/escalate).
\item Sends customer-facing welcome emails with tenant ID and login URL.
\item Generates HTML/Markdown/JSON reports for auditing and traceability.
\item Post-provisioning: monitors progress, detects risks, sends reminders, escalates stalled onboardings.
\end{enumerate}

{\scriptsize
\vspace{10pt}
\begin{itemize}
\item Salesforce is source of truth for accounts
\item CLM status reflects actual signatures
\item Invoice payment status is current
\item Invoices may be in different currencies (e.g., CAD, USD); cross-currency comparison uses historical ECB rates from the invoice date with a 2\% alignment threshold
\item One invoice per account per onboarding run
\item One Opportunity per Account for onboarding (no multi-deal scenarios)
\item Contract has a single version with no amendments
\item CS monitors Slack channels for alerts
\item Single onboarding per account at a time
\item Customer email available in CLM signatories
\item Account ID is consistent across all systems
\end{itemize}

\subsection{Limitations}
{\scriptsize
\vspace{10pt}
\begin{itemize}
\item \textbf{Mock integrations}: Salesforce, NetSuite, and CLM use simulated data; not connected to real APIs
\item \textbf{In-memory state}: Provisioning and task data lost on restart; no persistent database
\item \textbf{Single instance}: No horizontal scaling or load balancing for high-volume webhooks
\item \textbf{No approval UI}: ESCALATE decisions notify Slack but lack a UI for human approve/reject workflow
\item \textbf{No webhook auth}: Inbound webhooks not verified via HMAC signature or shared secret
\end{itemize}

\section{Multi-Agent Architecture via Model Context Protocol (MCP) Server Connections}

\begin{minipage}[t]{0.54\textwidth}
\vspace{0pt}
{\scriptsize\textit{Note: All 21 agent tools are mirrored as FastMCP server definitions in \texttt{app/mcp/} (7 servers: Salesforce, CLM, NetSuite, Currency, Provisioning, Notifications, Validation). These can be extracted to standalone MCP services for multi-agent collaboration.}}

\vspace{0.15em}
{\scriptsize
The architecture supports multi-agent collaboration through the \textbf{Model Context Protocol (MCP)}, enabling specialized agents to interact through standardized, governed tool interfaces rather than direct API access. Each integration is wrapped as a FastMCP server, cleanly separating decision logic from execution concerns while allowing agents to remain focused on their respective domains.
}

\vspace{0.15em}
\begin{table}[H]
\scriptsize
\begin{tabularx}{\textwidth}{@{}lX@{}}
\toprule
\textbf{Agent} & \textbf{Responsibility \& MCP Tools} \\
\midrule
Coordinator & Orchestrates end-to-end workflow (\texttt{salesforce.*}, \texttt{provision.*}) \\
Contract Agent & Monitors contract execution and signatures (\texttt{clm.get\_contract}) \\
Finance Agent & Evaluates billing, payments, and cross-currency alignment (\texttt{netsuite.*}, \texttt{currency.*}) \\
Task Monitor & Detects overdue onboarding tasks (\texttt{tasks.get\_overdue}) \\
\bottomrule
\end{tabularx}
\end{table}

\vspace{-15pt}
{\scriptsize\textbf{Benefits}: Independent versioning per integration $\bullet$ Clear domain ownership $\bullet$ Centralized access control and audit logging $\bullet$ Scalable agent complexity $\bullet$ Cross-agent context sharing.}

\vspace{0.35em}
{\scriptsize
\textbf{Concurrency \& API Execution Strategy}: MCP servers would act as controlled execution layers for external APIs, enabling concurrent and batch-aware request handling. Where supported, batch or composite API calls are leveraged to reduce network overhead and latency (e.g., Salesforce composite reads). If batch execution fails or yields partial results, the system safely falls back to bounded parallel single-entity requests.

Concurrency limits, rate limiting, and circuit breakers would be enforced at the MCP layer to prevent downstream system overload and cascading failures. This approach improves onboarding throughput while preserving idempotency, error isolation, and predictable failure handling across agents.
}

%-------------------------------
% Agent Runtime & API Governance
%-------------------------------
\vspace{0.3em}
{\small\bfseries\color{primary} Agentic Execution \& API Governance}

\vspace{0.1em}
{\scriptsize
\textbf{Deterministic Decision Boundaries}: The onboarding agent uses an agentic architecture where the LLM reasons through tools via native tool calling (Pydantic AI \texttt{@agent.tool} decorators). However, deterministic business rules enforced by \texttt{validate\_business\_rules} and \texttt{check\_financial\_alignment} tools define all violations and warnings. The LLM decides the routing (\texttt{PROCEED}, \texttt{ESCALATE}, \texttt{BLOCK}) based on these tool outputs, ensuring predictable, testable, and auditable outcomes.

\vspace{0.15em}
\textbf{Cross-System Data Chaining}: The agent fetches the Salesforce account first, then user/opportunity in parallel. Subsequent fetches chain sequentially: Opportunity.ContractId $\rightarrow$ SF Contract $\rightarrow$ CLM (via salesforce\_contract\_id) $\rightarrow$ Invoice (via clmContractRef). Fallback to account-based lookups if chain links are missing. For production, a tiered optimization strategy applies: \textbf{(1)} Batch API requests where supported (e.g., Salesforce Composite API); \textbf{(2)} Concurrent fetches for systems that don't support batching; \textbf{(3)} Bounded retry logic with exponential backoff (max 3 retries, 1s/2s/4s delays) for transient failures.

\vspace{0.15em}
\textbf{Idempotency \& Correlation-Based Execution}: Each onboarding run is keyed by a \texttt{correlation\_id} propagated across webhook ingestion, integrations, logs, reports, and notifications. This enables safe replays, deduplication of duplicate webhook events, and re-entrant execution without risk of duplicate provisioning, notifications, or side effects.

\vspace{0.15em}
\textbf{API Error Classification \& Retry Strategy}: External API failures are classified into retryable (e.g., transient 5xx, rate limiting) and non-retryable (e.g., validation, authorization) categories. Retryable errors follow bounded exponential backoff, while non-retryable errors immediately halt onboarding and surface actionable diagnostics. Circuit breakers prevent cascading failures across dependent systems.

\vspace{0.15em}
\textbf{Rate Limiting \& Backpressure}: Inbound webhook traffic is protected via per-source rate limits and queue-based ingestion to absorb traffic spikes from batch updates or retries. Concurrency limits are enforced per integration to prevent overwhelming Salesforce, CLM, or NetSuite APIs during peak load.

\vspace{0.15em}
\textbf{Agent Versioning \& Safe Rollouts}: Every onboarding run records the agent version, rule set, and prompt version used. This enables full auditability, regression analysis, and controlled rollouts (e.g., canary deployments) of agent logic and prompt updates without impacting in-flight executions.

\vspace{0.15em}
\textbf{LLM Data Governance \& Security}: All data sent to LLMs is field-filtered and sanitized. PII is masked prior to inference, and credentials are never exposed beyond integration boundaries. Trust boundaries are strictly enforced at the API layer.

\vspace{0.15em}
\textbf{Multi-Agent Interoperability}: All 21 agent tools are mirrored as FastMCP server definitions (7 servers in \texttt{app/mcp/}), separating agent logic from external system access. These can be extracted to standalone MCP services with independent versioning, access control, and audit logging. The A2A protocol provides a standardized layer for agent-to-agent collaboration without exposing internal state across boundaries.
