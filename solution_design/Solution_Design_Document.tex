\section{Architecture Overview}

The solution implements an \textbf{AI-powered automation agent} that orchestrates enterprise customer onboarding from deal closure through SaaS provisioning. Built with \textbf{Pydantic AI} for agentic tool calling and \textbf{FastAPI} for the REST interface, the agent uses native LLM tool calling to autonomously reason through 28 registered tools. Each tool is mirrored as a \textbf{FastMCP} server definition for MCP-style extraction. A second \textbf{CS Assistant Agent} shares all tools but returns free-form text for conversational interaction via a \textbf{Streamlit UI} with four pages (Dashboard, Portfolio Overview, Run Onboarding, Chat). Data flows sequentially across systems via \textbf{cross-system chaining}: Opportunity.ContractId $\rightarrow$ SF Contract.Id $\rightarrow$ CLM.salesforce\_contract\_id $\rightarrow$ Invoice.clmContractRef, with fallback to account-based lookups when chain links are missing. The latest prototype also closes the CS action loop: suggested actions can simulate upstream issue resolution in mocked SaaS systems and then re-run onboarding so a previously blocked or escalated account can actually progress into provisioning.

\section{AI Agent Application}

The agent leverages \textbf{OpenAI GPT-4o-mini} (configurable via \texttt{OPENAI\_MODEL}) with \textbf{Pydantic AI} for native tool calling. The LLM autonomously decides which of the 28 registered tools to call, in what order, and how many times. Falls back to \textbf{Ollama} local models when no API key is available. A second \textbf{CS Assistant Agent} shares all tools with free-form text output for conversational monitoring via the Streamlit UI.

\vspace{0.05em}
\begin{minipage}[t]{0.56\textwidth}
\vspace{0pt}
\subsection{LLM-Powered Intelligence}
\begin{itemize}
\item \textbf{Risk Analysis}: Evaluates API errors, violations, warnings $\rightarrow$ risk levels (low/medium/high/critical) with business impact and resolution time estimates.
\item \textbf{Summary Generation}: Human-readable reports for CS teams, translating technical states to actionable insights.
\item \textbf{Action Recommendations}: Prioritized steps with owner assignment (CS, Finance, IT/DevOps, Legal, Sales Ops).
\item \textbf{Error Interpretation}: Converts technical codes (e.g., \texttt{INVALID\_SESSION\_ID}) to plain English with resolution steps.
\end{itemize}

\vspace{2pt}

\subsection{Autonomous Actions}
\begin{enumerate}[label=\textbf{(\arabic*)}, itemsep=0pt, topsep=1pt]
\item Auto-provisions the SaaS tenant with tier-appropriate configuration (Enterprise/Growth/Starter).
\item Creates a 14-task onboarding checklist with dependencies, ownership (system/cs\_team/customer), and due dates.
\item Sends Slack notifications to \texttt{\#cs-onboarding-alerts} (blocked) or \texttt{\#cs-onboarding} (success/escalate).
\item Sends customer-facing welcome emails with tenant ID and login URL.
\item Generates HTML/Markdown/JSON reports for auditing and traceability.
\item Post-provisioning: monitors progress, detects risks, sends reminders, escalates stalled onboardings.
\item Portfolio management: aggregated alerts across accounts, suggested actions with approve/dismiss, batch reminders.
\item Remediation loop: the CS assistant can simulate source-system fixes (deal closed, contract executed, invoice paid, missing record created) and safely re-run onboarding from BLOCK/ESCALATE into PROCEED.
\item Sentiment analysis: ML-powered scoring of customer interactions via DistilBERT (PyTorch + HuggingFace) with trend detection; runs as a mandatory step (step 5) after every provisioning alongside risk assessment; negative sentiment shifts health to at\_risk and triggers proactive outreach suggestions.
\item Product assistance: looks up tier configuration (features, limits, storage), SLA details, contractual terms, and implementation prerequisites from CLM and provisioning data.
\end{enumerate}

\begin{itemize}[leftmargin=1.2em]

\item \texttt{GET /tasks/\{id\}/pending?owner=cs\_team} -- Filtered pending
\item \texttt{GET /tasks/\{id\}/overdue} -- Overdue task alerts
\item \texttt{PUT /tasks/\{id\}/\{task\_id\}} -- Update task status
\item \texttt{GET /tasks/\{id\}/next-actions} -- Ready-to-execute tasks
\end{itemize}
\end{minipage}

\vspace{0.1em}

\vspace{0.1em}
\subsection{Sentiment Analysis as a Proactive Risk Signal}
{\scriptsize
Beyond task and billing state, the CS assistant also tracks \textbf{customer sentiment} from inbound and outbound interactions such as email, chat, support tickets, and calls. Sentiment scoring uses a \textbf{DistilBERT transformer model} (\texttt{distilbert-base-uncased-finetuned-sst-2-english}) via \textbf{PyTorch + HuggingFace Transformers} for real ML inference, with graceful fallback to keyword-based scoring when the ML stack is not installed. The model is lazy-loaded on first use to avoid impacting startup time. The prototype exposes two sentiment tools: \texttt{log\_customer\_interaction} records structured interactions for an account, and \texttt{get\_customer\_sentiment} returns a score, label (positive/neutral/negative), trend (improving/stable/declining), and the model used. This allows the agent to identify onboarding risk \emph{before} a task is formally overdue or blocked.
}

\vspace{0.08em}
\begin{table}[H]
\scriptsize
\begin{tabularx}{\textwidth}{@{}lX@{}}
\toprule
\textbf{Sentiment Signal} & \textbf{Agent Behavior} \\
\midrule
Negative sentiment + declining trend & Recommend proactive outreach, schedule a check-in call, raise account priority in portfolio alerts \\
Negative sentiment + blocked onboarding tasks & Escalate urgency because operational blockers and customer frustration are compounding \\
Improving sentiment after outreach & Reduce risk posture and keep onboarding in the monitored PROCEED path \\
\bottomrule
\end{tabularx}
\end{table}

{\scriptsize
\vspace{6pt}
\begin{itemize}

\item CLM status reflects actual signatures
\item Invoice payment status is current
\item Cross-currency comparison uses historical ECB rates from invoice date with 2\% threshold
\item One invoice per account per onboarding run
\item One Opportunity per Account (no multi-deal)
\item Contract has a single version, no amendments
\item CS monitors Slack channels for alerts
\item Single onboarding per account at a time
\item Customer email available in CLM signatories
\item Account ID consistent across all systems
\end{itemize}

\begin{minipage}[t]{0.32\textwidth}
{\scriptsize
\subsection{Limitations}
{\scriptsize
\vspace{6pt}
\begin{itemize}
\item \textbf{Mock integrations}: Salesforce, NetSuite, CLM use simulated data; not connected to real APIs
\item \textbf{In-memory state}: Provisioning and task data lost on restart; no persistent database
\item \textbf{Single instance}: No horizontal scaling or load balancing
\item \textbf{Limited approval UI}: Suggested Actions panel provides approve/dismiss and real backend execution; full Slack interactive approval with identity/policy controls not yet implemented
\item \textbf{No webhook auth}: Inbound webhooks not verified via HMAC or shared secret
\end{itemize}
}

\section{Multi-Agent Architecture via Model Context Protocol (MCP) Server Connections}

\begin{minipage}[t]{0.54\textwidth}
\vspace{0pt}
{\scriptsize\textit{Note: All 28 agent tools are mirrored as FastMCP server definitions in \texttt{app/mcp/} (8 servers: Salesforce, CLM, NetSuite, Currency, Provisioning, Notifications, Validation, Sentiment). These can be extracted to standalone MCP services for multi-agent collaboration.}}

\vspace{0.1em}
{\scriptsize
The architecture supports multi-agent collaboration through the \textbf{Model Context Protocol (MCP)}, enabling specialized agents to interact through standardized, governed tool interfaces rather than direct API access. Each integration is wrapped as a FastMCP server, cleanly separating decision logic from execution concerns.
}

\vspace{0.1em}
\begin{table}[H]
\scriptsize
\begin{tabularx}{\textwidth}{@{}lX@{}}
\toprule
\textbf{Agent} & \textbf{Responsibility \& MCP Tools} \\
\midrule
Coordinator & Orchestrates end-to-end workflow (\texttt{salesforce.*}, \texttt{provision.*}) \\
Contract Agent & Monitors contract execution and signatures (\texttt{clm.get\_contract}) \\
Finance Agent & Evaluates billing, payments, and cross-currency alignment (\texttt{netsuite.*}, \texttt{currency.*}) \\
Task Monitor & Detects overdue onboarding tasks (\texttt{tasks.get\_overdue}) \\
Sentiment Agent & Scores customer interactions via ML inference (\texttt{sentiment.*}) \\
\bottomrule
\end{tabularx}
\end{table}

\vspace{-15pt}
{\scriptsize\textbf{Benefits}: Independent versioning per integration $\bullet$ Clear domain ownership $\bullet$ Centralized access control and audit logging $\bullet$ Scalable agent complexity $\bullet$ Cross-agent context sharing.}

\vspace{0.1em}
{\scriptsize
\textbf{Concurrency \& API Execution Strategy}: MCP servers act as controlled execution layers for external APIs, enabling concurrent and batch-aware request handling. Batch or composite API calls are leveraged where supported (e.g., Salesforce composite reads), with fallback to bounded parallel single-entity requests. Concurrency limits, rate limiting, and circuit breakers are enforced at the MCP layer to prevent downstream overload and cascading failures.
}
